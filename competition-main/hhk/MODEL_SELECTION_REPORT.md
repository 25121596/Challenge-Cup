## 二、内存占用 (memcheck.py)

| 配置 | IQ4_XS | Q8_0 |
|---|---|---|
| 模型文件 | 1.09 GB | 2.01 GB |
| ctx=3072, kv=q8_0, ngl=0 | **1.53 GB** ❌ | 未测 (超预算必然) |
| ctx=3840, kv=q4_0, ngl=0 | **1.46 GB** ✅ | 未测 (超预算必然) |
| CPU 推理速度 (ngl=0) | **59.5 t/s** | 未测 |
| GPU 推理速度 (ngl=99) | ~355 t/s | ~357 t/s |
| 边缘预算 (1.5GB) | ✅ **达标** (kv=q4_0 时) | ❌ 文件即超预算 |

### 关键发现

1. IQ4_XS 在 kv_cache=q4_0 + ctx=3840 下恰好卡在 1.5GB 以内，剩余仅 ~40MB 余量
2. kv_cache 选 q8_0 会超出预算 (1.53GB)，**部署时必须用 q4_0**
3. Q8_0 模型文件就 2.01GB，完全不适合边缘设备，直接淘汰

### 建议配置

```
模型: Qwen3-1.7B-IQ4_XS
KV Cache: q4_0
Context: 2048 (生产环境, 留 200MB 安全余量)
ngl: 0 (纯 CPU)
预估内存: ~1.3 GB (含 runtime 开销)
```

## 四、端到端场景实测 (scenario_runner.py)

测试场景: **工业设备状态监测与故障诊断** (industrial_inspection)
数据: Nature 三相电机 edge_sim 子集 (332条)
测试用例: 正常 / 轴承磨损 / 轴承失效 / 电流不平衡 / 过热

| 指标 | 实测值 | 赛题要求 | 达标 |
|---|---|---|---|
| 成功率 | **100.0%** | ≥90% | ✅ |
| 平均时延 | 3679.8 ms | ≤200ms | ❌ |
| P50 时延 | 3675.3 ms | — | — |
| P95 时延 | 3730.4 ms | ≤200ms | ❌ |
| 200ms 达标率 | **0.0%** | ≥95% | ❌ |
| 甩云率 | 100.0% | ≤30% | ❌ |
| 生成速度 | 55.6 t/s | — | — |

### 时延分析

3.7秒延迟的构成:
```
prompt 处理: ~18ms
生成 200 tokens: 200 / 55.6tps ≈ 3.6s
总延迟 ≈ 3.7s
```

**根本原因**: ngl=0 (纯 CPU) + n_predict=200 tokens。每 token 约 18ms，200 token = 3.6s。

### 甩云分析

甩云率 100% 的原因:
- 模型输出含 markdown 代码块标记 (```json)，`parse_response()` 解析失败 → 判为"格式异常" → 甩云
- 时延远超 200ms 阈值

**可优化**: 限制 n_predict=80、用 GBNF grammar 约束 JSON 输出、GPU offload 若干层

## 五、最终选型结论

### ✅ 推荐: Qwen3-1.7B-IQ4_XS

| 评估维度 | 结论 |
|---|---|
| 内存 | **唯一能在 1.5GB 内运行的选项** (Q8_0 文件就超了) |
| 工业能力 | 安全/数据/双语三项优于 Q8_0，更适合比赛场景 |
| 速度 | 60 t/s CPU，350+ t/s GPU，满足推理需求 |
| 短板 | 数学推理弱(30.8)、语言推理弱(45.3) — 可甩云弥补 |

### 选型策略

```
边缘端 (IQ4_XS):
  - 实时监测、故障初筛、简单分类
  - n_predict ≤ 80, 目标时延 ≤ 1.5s
  - 加 GBNF grammar 强制 JSON 输出

云端 (Qwen3-32B / Qwen2.5-72B):
  - 复杂诊断、数学推理、长文本分析
  - 边端甩云任务二次确认
  - 模型更新 + LoRA 调优
```
